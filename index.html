üïê Understanding SCD2 Complexity

Slowly Changing Dimensions Type 2 tracks historical changes. Here's why it's challenging:

‚ùå The Problem: Overlapping Windows

Customer_ID | Name | Valid_From | Valid_To | Is_Current
101 | John | 2023-01-01 | 2023-06-30 | N
101 | John | 2023-06-15 | 9999-12-31 | Y ‚Üê OVERLAP!

Issue: Records overlap between June 15-30. Which version is correct?

‚úÖ Correct SCD2 Structure

Customer_ID | Name | Valid_From | Valid_To | Is_Current
101 | John | 2023-01-01 | 2023-06-14 | N
101 | John D. | 2023-06-15 | 9999-12-31 | Y ‚úì No gaps or overlaps

‚ö†Ô∏è Common SCD2 Defects We Catch

  • Multiple records marked as current (is_current = Y)
  • Overlapping date ranges
  • Gaps in historical timeline
  • Invalid future dates in valid_from

‚ö° Agile Data QA Framework (A‚ÄìH)

A lightweight, deterministic framework designed for rapid audits and repeatability.

A. Scope Validation ‚Äì Confirm dataset boundaries
B. Core Metrics Review ‚Äì Row counts and sanity checks
C. Duplicate Current Records ‚Äì Detect multiple active records
D. Temporal Integrity (SCD2) ‚Äì Validate non-overlapping windows
E. Mandatory Attributes Check ‚Äì Critical fields populated
F. Date & Format Validation ‚Äì Parsability and consistency
G. Encoding & Character Hygiene ‚Äì No invalid characters
H. Referential Integrity ‚Äì Relationships remain intact

üíª Framework in Action: SQL Examples

Step C: Duplicate Current Records

-- Find entities with multiple active records
SELECT business_key, COUNT(*) as active_count
FROM customer_dim
WHERE is_current = TRUE
GROUP BY business_key
HAVING COUNT(*) > 1;

-- Expected result: 0 rows

Step D: SCD2 Temporal Integrity

-- Detect overlapping date windows
SELECT a.business_key, a.valid_from, a.valid_to,
b.valid_from, b.valid_to
FROM customer_dim a
JOIN customer_dim b ON a.business_key = b.business_key
WHERE a.valid_from < b.valid_to
AND b.valid_from < a.valid_to
AND a.surrogate_key != b.surrogate_key;

-- Expected result: 0 rows

Step E: Mandatory Attributes

-- Check for nulls in critical fields
SELECT COUNT(*) as null_violations
FROM customer_dim
WHERE email IS NULL
OR first_name IS NULL
OR customer_type IS NULL;

‚ö†Ô∏è Common Pitfalls & How to Avoid Them

‚ùå Pitfall #1: Testing in Production Only

Problem: Catching defects after data is already published

Solution: Implement validation in staging environment first

‚ùå Pitfall #2: Over-relying on Sampling

Problem: Critical defects exist in non-sampled data

Solution: Always run full-scope checks on critical business rules (duplicates, SCD2)

‚ùå Pitfall #3: No Audit Trail

Problem: Can't prove what was validated when issues arise

Solution: Store validation results with timestamps and row counts

‚ùå Pitfall #4: Ignoring "Small" Issues

Problem: 0.1% null rate seems acceptable, but = 1,000 bad records

Solution: Set absolute thresholds, not just percentages
